# AI Context

Use this file when an LLM is asked to add tests, update support code, or maintain this framework.

## What this project is

This is a generated `Cypress` + `TypeScript` automation framework.

It is intentionally small and readable:

- specs describe workflows and assertions
- support commands and page modules own selectors and reusable actions
- `DataFactory` creates repeatable test data
- the bundled UI demo app exists so the scaffold works immediately after install

## Architecture map

- `cypress/e2e/`
  - spec files and assertions
- `cypress/support/commands.ts`
  - shared Cypress commands
- `cypress/support/pages/`
  - page modules that own selectors and page-level actions
- `cypress/support/data/`
  - generic test data builders
- `config/`
  - runtime env, secrets, and base URLs
- `demo-apps/`
  - local sample app for starter runs

## Authoring rules

- Keep selectors out of spec files.
- Put selectors in support page modules or commands.
- Keep assertions in spec files.
- Do not use arbitrary waits.
- Prefer stable selectors such as labels, button text, and `data-testid`.
- Use `DataFactory` for repeatable sample data.
- Prefer adding Cypress commands for common workflows rather than large helper classes.

## Local run model

- `npm test` runs headless Cypress.
- `npm run open` launches the interactive runner.
- In local `dev`, the bundled sample app auto-starts when the default local URL is in use.
- Local credentials are generated into `.env` on first run.

## Common tasks

### Add a new journey

1. Add or extend a page module in `cypress/support/pages/`.
2. Add or extend a command in `cypress/support/commands.ts` if the workflow is reused.
3. Add the spec under `cypress/e2e/`.
4. Keep assertions in the spec.

### Fix a broken selector

1. Update the page module or command.
2. Do not patch the spec with direct selectors.
3. Keep selectors semantic and stable.

### Add reusable behavior

1. Use a custom Cypress command for repeated workflows.
2. Use a page module for page-specific locators and helpers.
3. Keep the command API business-friendly.

## Safe changes

- Add specs
- Extend commands or page modules
- Add generic data helpers
- Adjust config and env wiring carefully

## Changes to avoid casually

- Do not move selectors into specs.
- Do not add brittle waits.
- Do not make commands huge and stateful without a reason.
- Do not treat the sample app as product architecture.

## Upgrade guidance

If the framework was generated by `testkit`, prefer:

```bash
npx -y @toolstackhq/create-testkit upgrade check .
npx -y @toolstackhq/create-testkit upgrade apply --safe .
```

Safe upgrades are for managed framework files, not user-owned spec logic.
