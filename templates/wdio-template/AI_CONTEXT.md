# AI Context

Use this file when an LLM is asked to add specs, update page objects, or maintain this framework.

## What this project is

This is a generated `WebdriverIO` + `TypeScript` automation framework.

It is designed to stay understandable:

- specs describe the workflow and assertions
- page objects own selectors and browser actions
- `DataFactory` creates repeatable sample data
- the bundled sample app exists so the framework runs immediately after install

## Architecture map

- `tests/`
  - Mocha specs and assertions
- `pages/`
  - selectors and user actions
- `components/`
  - reusable UI helpers when multiple pages share behavior
- `data/`
  - generic test data builders
- `config/`
  - environment selection and secrets
- `reporters/`
  - structured logs and reporting helpers
- `demo-apps/`
  - bundled UI sample app

## Authoring rules

- Keep selectors out of spec files.
- Put selectors in page objects.
- Keep assertions in spec files.
- Do not use `browser.pause`.
- Prefer semantic selectors and `data-testid`.
- Use `DataFactory` instead of ad hoc random data.
- Extend existing page objects before creating new overlapping ones.

## Local run model

- `npm test` runs the WDIO suite.
- In local `dev`, the bundled sample app auto-starts when the default local URL is in use.
- Local credentials are generated into `.env` on first run.
- Reports land under `reports/`, `allure-results/`, and `test-results/`.

## Common tasks

### Add a new spec

1. Add or extend a page object in `pages/`.
2. Add the spec under `tests/`.
3. Create sample data with `DataFactory`.
4. Assert in the spec, not the page object.

### Fix a broken selector

1. Update the page object.
2. Do not patch the spec with direct `$()` calls.
3. Keep selector intent readable.

### Add reusable behavior

1. Use page-object methods for page-specific workflows.
2. Use `components/` only if multiple pages share the same widget behavior.
3. Keep helper names user-facing, not DOM-facing.

## Safe changes

- Add specs
- Extend page objects
- Add data builders
- Adjust env/config wiring carefully
- Extend reporting if defaults remain usable

## Changes to avoid casually

- Do not put selectors in specs.
- Do not use `browser.pause`.
- Do not move assertions into page objects.
- Do not treat the sample app as real product architecture.

